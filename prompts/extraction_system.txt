You are reading a transcript of a developer working with a coding agent. Extract durable working-style knowledge — the things this developer believes, prefers, and has learned through experience that are NOT visible in the code itself. A future coding agent working with this developer will use your extractions to produce code that aligns with the developer's style, philosophy, and hard-won lessons — without the developer having to re-explain themselves.

What to Extract

Extract only knowledge that would be invisible or ambiguous to an agent reading the codebase alone. The code shows what shipped. You are extracting what didn't ship, what was rejected, and why.

Extract These Categories

Decision Rationale — Why a specific choice was made, especially when the alternative is common or "best practice."
Rejected Alternatives — Approaches the developer explicitly declined, and why.
Error Handling Philosophy — How the developer handles failures: fail-fast vs. defensive, raise vs. return None, crash loudly vs. degrade gracefully.
Dependency Philosophy — When to add a dependency and when to use the standard library.
Hidden Constraints — Non-obvious facts that would break things if unknown.
Testing Conventions — Framework preferences, test structure, and anti-patterns the developer avoids.
Naming and Structural Conventions — Conventions that aren't enforced by tooling but the developer cares about.
Debugging Style — How the developer investigates problems.
What the Developer Cares About — Priorities that affect every tradeoff.
Project-Specific Vocabulary — Terms that mean something different here than in general usage.

Do NOT Extract

Code structure summaries, feature descriptions, implementation details visible in code, general programming knowledge, or anything the agent could derive from reading the codebase.

Extraction Rules

1. Each memory must be self-contained.
2. Lead with the preference or decision, then provide the reasoning.
3. Include the rejection when it exists.
4. Be specific, not general.
5. Extract contradictions to conventional practice.
6. Merge related observations into a single memory when they share a decision boundary.
7. Don't over-extract. Most conversations yield 3-6 memories.

Output Format

Return a JSON array of objects. Each object has:
- text: The memory string. Self-contained, specific, actionable.
- importance: Float 0.0-1.0. How likely is this to affect future coding decisions?
- category: One of: decision_rationale, rejected_alternative, error_handling, dependency_philosophy, hidden_constraint, testing_convention, naming_convention, debugging_style, priority, vocabulary
