You are evaluating a cluster of developer memories that are semantically similar. Decide whether they should be squashed into a single consolidated memory or kept as separate memories. This runs as a periodic maintenance operation — there is no urgency, no user waiting, and no cost to being conservative.

What You Receive

A cluster of 2-8 memory texts, each a self-contained statement about a developer's working-style knowledge extracted from coding conversations. These memories are similar enough (cosine >= 0.85) to be clustered together, but similarity does not mean redundancy.

Decision Rules

Squash when:
- The memories express the same decision or preference with different wording or detail levels.
- One memory subsumes another.
- The memories are the same preference observed in different sessions — re-extraction produces near-duplicates.

Keep separate when:
- The memories address different decisions, even if they reflect the same underlying philosophy.
- One memory has project-specific context that the other lacks.
- Squashing would lose specificity.

Consolidation Rules

1. Preserve the most specific phrasing.
2. Merge reasoning, don't average it.
3. Drop nothing that isn't subsumed.
4. The consolidated text must be self-contained.
5. One memory per decision boundary.

Output Format

Return a JSON object:
{
  "decision": "squash" | "separate",
  "consolidated": "The consolidated memory text (only if decision is squash)",
  "reasoning": "Brief explanation of why you chose squash or separate"
}

If decision is "separate", consolidated should be null.

Examples

Example 1: Squash (same decision, different detail)

Cluster:
- "Prefers threading over asyncio for concurrent I/O"
- "Rejected asyncio for the task runner, called it a mess — use threading instead"
- "Prefers threading.Event and Timer patterns for concurrency, not async/await"

Output:
{"decision": "squash", "consolidated": "Prefers threading over asyncio for concurrent I/O — explicitly rejected asyncio after painful prior experience, uses threading.Event and Timer patterns", "reasoning": "All three memories describe the same decision (threading over asyncio) with increasing specificity."}

Example 2: Keep separate (different decisions, shared philosophy)

Cluster:
- "Prefers raw sqlite3 over ORM for single-table CRUD — avoids dependency overhead when queries don't justify the indirection ceremony"
- "Prefers threading over asyncio for concurrent I/O — explicitly rejected asyncio after a prior bad experience"

Output:
{"decision": "separate", "consolidated": null, "reasoning": "Both reflect dependency minimalism but address different technical decisions. Squashing would require abstracting to 'prefers simple solutions' which is too vague."}

Example 3: Keep separate (different categories)

Cluster:
- "Tests with timing dependencies must use synchronization primitives (Event.wait), not sleep"
- "Prefers debug-level logging for retry/polling noise, not info-level"

Output:
{"decision": "separate", "consolidated": null, "reasoning": "Both relate to testing/reliability conventions but address completely different concerns."}
